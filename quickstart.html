Quickstart

Zero → first approved flow in about 10 minutes. You will install RailCall, launch the local Studio, connect one connector with your own key, stage a write (and see that nothing has been written), approve it through the saga, and verify the signed receipt. Everything runs on your machine.

1Install~1 min

bash
curl -fsSL https://railcall.ai/install.sh | bash

That installs the CLI and the Studio. No browser download, no Gatekeeper prompt. Updates ship the same way — re-run the one-liner.

2Launch the Studio~1 min

bash
railcall studio

The Studio opens in your browser on 127.0.0.1:8799 (override with STUDIO_PORT). It binds to loopback only — nothing is exposed to your network — and every mutating endpoint requires a per-launch session token, so a random web page can’t drive it.

Describe → build works out of the box. The Builder’s chat runs on the hosted engine by default — railcall login <your key> authenticates it and each call books one governed flow (free accounts include 500). Paste your own model key in Connect to run fully local instead; a local key always wins. Your key, flows, and balance live in the customer portal.

3Connect a connector~2 min

Open the INTEGRATE tab, pick a connector, and paste your key. Keys vault to a local, owner-only (0600) file on your machine — they never leave it, and they are only ever sent to the provider’s own API, in live mode. Malformed keys are refused before any socket opens (e.g. GitHub tokens must look like a PAT; Stripe accepts sk_test_* only and refuses live-mode keys outright).

Full key tables, write actions and rollback semantics per connector: Connectors.

No key yet? Use mock mode. Every connector has a mock switch: set its *_API_BASE vault key (Discord and the webhook connectors: *_MOCK_BASE) to a loopback URL, e.g. AIRTABLE_API_BASE = http://127.0.0.1:8802. The entire stage→approve→receipt flow below then runs with zero external calls and no token on the wire — and the receipt stays honest about it: mode: "mock", external_api_touched: false. A mock run never claims a real touch.

4Open RUN LIVE and stage~2 min

Open the connector’s RUN LIVE page — e.g. 127.0.0.1:8799/run_github, /run_sheets, /run_airtable, /run_stripe… The page header carries the approval-gated badge.

Click the stage button. While it runs the status reads ⏳ staging (dry-run, nothing sent)... — then the staged plan appears under PENDING · review before approving for you to read before anything happens.

Look at the status line. On the Airtable and Sheets pages it reads like:

✓ 3 change(s) staged signed delta nothing written yet

(single-action pages such as GitHub, Notion, Slack, Discord and the webhooks phrase the last part nothing sent yet).

That amber phrase is literal. Staging computed the plan, hashed it (sha256), signed it (Ed25519), and wrote a one-time staging file locally — and did zero writes to the external system. Close the tab now and nothing ever happens.

5Approve~1 min

Click the approve button — ✓ approve & apply on Airtable/Sheets (Stripe spells out the stakes: ✓ approve & charge $…; the single-action pages name the action, e.g. opening the issue). This is the airlock crossing:

  • The engine re-computes the staged delta’s integrity hash and re-verifies its signature. Any mismatch refuses to execute.
  • It then fires exactly the staged delta — no more, no less — through the saga, with the connector’s rollback compensator wired (restore the original row, close the issue, refund the intent, delete the message, archive the page, or fire the honest .cancel counter-event for webhooks).
  • If any step fails, the compensator runs automatically and the result shows ROLLED_BACK with the compensator’s note, instead of OK.
  • A staged delta is one-time use — once executed it cannot be re-fired.

Want to watch the rollback without waiting for a real failure? Tick 🔥 inject failure (rollback demo) before approving. The write executes, an injected failure follows, the compensator undoes it, and the results header flips from APPLIED to ROLLED BACK. The receipt records the demo honestly as force_fail_demo: true.

6Verify the signed receipt~2 min

After approval the status line shows the leading characters of the receipt’s hash — ✓ done · receipt sha256:… — and the approve response carries the full receipt JSON. Things worth checking:

fieldexpect
mode"live" or "mock" — bound when you staged, not when you approved.
external_api_touchedEquals mode == "live". Your mock trial says false.
force_fail_demotrue only if you ticked the rollback demo.
approval.approved_integrityMatches delta_integrity — the exact delta you reviewed is the one that ran.
results[]One row per action: OK, ROLLED_BACK or ERROR, plus the compensator note on rollback.
integrity_hashsha256:… over the receipt body itself.
signatureEd25519 block — {"alg": "ed25519", "sig": …, "key_id": …}. If the local signing layer is unavailable the UI says unsigned rather than faking one; the sha256 integrity hash still binds the receipt.

And verify in the real world: the issue is open on your repo (or closed, if you ran the rollback demo — GitHub keeps it visible), the row changed in your sheet, the $1 test intent sits in your Stripe test dashboard with its refund next to it, the message landed in your channel.

That’s the whole model. Agents and workflows can plan anything; they cannot write to the outside world except through this gate — stage with zero writes, human approves, execute exactly the staged delta through a saga with rollback, receipt signed. The receipt is the trust layer.
