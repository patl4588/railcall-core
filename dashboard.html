Governance model
DRY-RUN first
Nothing executes until you approve it — the airlock gates every flow.
Limited outbound
Outbound is limited to the blind billing pulse (SHA-256 key hash + nonce + action) and — on hosted builds — the plain-English description you type (zero-retention). Your records, provider keys, and receipts never leave your machine.
Human approval — the REQUESTS queue
The approval queue lives in your local Studio, not this console: REQUESTS is the first item in the Studio sidebar. A live send flagged require_human holds there until you enter a terminal-only approve code — and a freeze kill-switch in the Studio header stops everything instantly.
Verify it yourself — no trust required
Every governed flow mints an Ed25519-signed receipt recording external_sockets: 0 during local processing — measured by an lsof sweep on your hardware, not asserted. Verify any receipt offline — the public key travels inside the receipt.
railcall verify <receipt>

Connect a tool — local adapters, no cloud relayer
RailCall is bring-your-own-key and local-first: a small set of generic local adapters takes your data straight from your machine to each provider — through the Airlock, receipted, with no third-party cloud in the middle (aggregator rails are the labeled exception). You set keys in the local Studio → Integrate, never on this page — each lands in a 0600 vault on your machine, and a connector shows tested only after a real key passes a live probe and mints a signed receipt (key-present otherwise — never fake-tested).
