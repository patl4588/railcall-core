System architecture

Almost nothing
leaves your machine.

RailCall runs the mechanical work — file operations, analysis, testing, workflow execution — locally on hardware you already own. The only things that ever cross your machine's boundary are reasoning tokens to a model you chose, and a run-count for billing. Your keys, your source, and your records stay put. Here is the exact shape of that.

Data plane · 127.0.0.1 2 outbound edges, both auditable Ed25519 receipts · verify offline

FIG.02

The governed loop

Every irreversible action — a file write, a shipped change, a payment, a message — passes all four gates in order. There is no path that skips one.

plan → approve → prove
GATE 01

BYOK vault

Keys resolve locally at 127.0.0.1. Nothing authenticates over a network you don't own.

→
GATE 02

Dry-run airlock

The full effect is computed and signed first. You see the exact diff before a single write.

→
GATE 03

Human approval

Keypair-based, not a login. A second terminal-only credential the browser never sees.

→
GATE 04

Signed receipt

The approved action mints an Ed25519 receipt with an integrity hash and full action log.

🔒No bypass. A missing signature, a stale plan, or a global freeze all fail closed — nothing fires.

Where the protocol goes next

Today, receipts are signed and verified locally. The next arc turns that private proof into a shared one — so an auditor, a partner, or another org's agent can trust a RailCall receipt without trusting the machine that made it. This is roadmap, stated as roadmap.

Next

Cross-org verification

Hand a receipt to a third party; they verify it against a shared trust registry, not your word.

Next

Governed audit ledger

Append-only, independently attested record of governed actions across machines.
